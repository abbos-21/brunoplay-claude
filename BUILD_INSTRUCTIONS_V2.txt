SECTION 2: PROJECT ARCHITECTURE
================================================================================

ARCHITECTURAL RULES (NON-NEGOTIABLE):

  RULE 1 — THIN ROUTES, FAT SERVICES
    Route files ONLY do: parse/validate request → call service → return response.
    Route files NEVER contain: database queries, business logic, calculations,
    or side effects.

  RULE 2 — PURE COMPUTATION FUNCTIONS
    All mathematical/deterministic logic lives in pure function files
    (*.engine.ts) with ZERO database or I/O imports. These must be 100%
    unit-testable.

  RULE 3 — TRANSACTIONS FOR MULTI-STEP MUTATIONS
    Any operation with 2+ database writes MUST use prisma.$transaction().

  RULE 4 — CENTRALIZED ERROR HANDLING
    Custom error classes in lib/errors.ts. Global Fastify error handler returns
    consistent JSON: { success: false, error: { code, message } }.

  RULE 5 — EVERY ENDPOINT VALIDATES INPUT
    Zod schemas for every request body, URL param, and query param.

  RULE 6 — SETTINGS ARE DYNAMIC, NOT HARDCODED
    All tunable game parameters load from the Settings DB table at runtime
    with 60-second in-memory cache. Hardcode ONLY structural invariants
    (FISHING_DURATION_MS = 3_600_000, MAX_LEVEL = 13, COMBO_ITEM_COUNT = 8,
    COMBO_PICK_COUNT = 4, JWT_EXPIRY = "7d").

SECTION 4: AUTHENTICATION & AUTHORIZATION
================================================================================

4.1  USER AUTHENTICATION FLOW

  1. Frontend obtains initData from Telegram Mini App SDK.
  2. Client sends POST /api/auth with { initData, ref? }.
  3. Server validates initData signature via HMAC-SHA256 using BOT_TOKEN
     (library: @tma.js/init-data-node). Validation window: 24 hours.
  4. Server extracts: telegramId, username, firstName, lastName, languageCode,
     isBot.
  5. Upsert user: create if not exists, update profile fields if exists.
  6. If ref parameter and user is NEW:
     - Parse ref: format is "ref_<userId>" or raw "<telegramId>"
     - Set referredById on the new user
  7. Update lastLoginAt to now().
  8. Return JWT token (signed with JWT_SECRET, expires 7 days).
  9. All subsequent requests: Authorization: Bearer <token>.

4.2  USER JWT MIDDLEWARE (Fastify plugin)

  - Extract token from Authorization header.
  - Verify JWT with JWT_SECRET.
  - Load User from DB by decoded id.
  - Reject blocked users (403).
  - Decorate request with user object.

4.3  ADMIN AUTHENTICATION

  - Registration: POST /api/admin/auth/register — requires SUPER_ADMIN JWT.
  - Login: POST /api/admin/auth/login — username + password (bcrypt verify).
  - Admin JWT signed with ADMIN_JWT_SECRET (7-day expiry).
  - Admin middleware verifies token, loads Admin record, checks role.

4.4  API ENDPOINTS

  POST  /api/auth
    Body: { initData: string, ref?: string }
    Response: { token: string, user: UserProfile, isNew: boolean }

SECTION 5: FISHING SYSTEM (CORE GAMEPLAY)
================================================================================

The Fishing section is the MAIN gameplay loop. Players earn coins by fishing.

5.1  MECHANICS

  - Player presses "Start Fishing" button.
  - Bruno performs a fishing animation (frontend) in a river.
  - Fishing lasts exactly 1 hour (3600 seconds, configurable in Settings).
  - During fishing, coins accumulate in the Coin Capacity Tank at the
    user's miningRate (coins per second).
  - The tank has a maximum capacity (tankCapacity). Coins stop accumulating
    when the tank is full.
  - After the hour completes (or at any time), the player presses the
    Fishing button again to COLLECT — coins transfer from tank to Main Balance.
  - If the player collects before the hour is up, they get whatever has
    accumulated so far.

5.2  BOOSTER EFFECTS ON FISHING

  - If the user has an active MINING_BOOST_1_5X booster:
    Effective mining rate = miningRate * 1.5
  - If the user has an active AUTO_MINING booster:
    Fishing automatically starts again after collection without user input.
    The server must handle this: when auto-mining is active and the user
    collects (or the tank fills), a new fishing session begins immediately.

5.3  COIN GENERATION FORMULA

  On sync/collect, calculate:

    elapsedSeconds = min(
      (now - fishingStartedAt) / 1000,
      fishingDurationSeconds           // cap at 1 hour
    )

    effectiveRate = miningRate * boosterMultiplier  // 1.0 if no booster

    coinsGenerated = min(
      elapsedSeconds * effectiveRate,
      tankCapacity - tankCoins          // don't exceed tank
    )

    newTankCoins = tankCoins + coinsGenerated

5.4  PURE FUNCTIONS (fishing.engine.ts)

  interface FishingState {
    isFishing: boolean;
    fishingStartedAt: Date | null;
    tankCoins: number;
    tankCapacity: number;
    miningRate: number;

  }

  interface FishingCalcResult {
    newTankCoins: number;
    coinsGenerated: number;

    fishingComplete: boolean;  // true if 1 hour has elapsed
  }

  function calculateFishingSync(
    state: FishingState,
    now: Date,

  function canStartFishing(state: FishingState): {
    allowed: boolean;
    reason?: string;

  Write comprehensive unit tests for edge cases:
    - 0 seconds elapsed
    - Exactly 3600 seconds elapsed
    - >3600 seconds elapsed (should cap)
    - Tank already full
    - Booster multiplier applied correctly
    - Tiny fractional rates

SECTION 6: COIN & WITHDRAWAL SYSTEM
================================================================================

6.1  COIN STATES

  Coin Capacity Tank (tankCoins):
    - Temporary storage during fishing.
    - Transferred to Main Balance on collection.
    - NOT directly spendable.

  Main Balance (coins):
    - Permanent spendable balance.
    - Used for withdrawals.
    - Cannot go negative.

6.2  WITHDRAWAL RULES

  - Exchange rate: 100,000 coins = 1 TON (configurable in Settings).
  - Minimum withdrawal: 100,000 coins (configurable).
  - Maximum withdrawal: 1,000,000 coins (configurable).
  - Blockchain fee: 0.05 TON (display only — deducted from the TON sent).
  - User must provide a valid TON wallet address.
  - Blocked users cannot withdraw.

6.3  WITHDRAWAL FLOW (TRANSACTIONAL)

  Inside a single prisma.$transaction():
    1. Re-read user (fresh state inside transaction).
    2. Validate: not blocked, sufficient balance, within min/max limits.
    3. Validate TON address format.
    4. Deduct coins from user.coins.
    5. Create Withdrawal record (status=PENDING).
    6. Create Action log entry.
  End transaction.

  Then (outside transaction):
    7. Queue TON transaction via tonService (p-queue, concurrency=1).
    8. On success: Update withdrawal status=COMPLETED, set txHash.
    9. On failure: Update withdrawal status=FAILED, set errorMessage.
       Do NOT refund automatically (admin must review failed withdrawals).

6.4  TRANSACTION HISTORY

  Return list of user's withdrawals with: address, date, time, amount in TON,
  status, txHash.

6.5  API ENDPOINTS

    Response: { coinToTonRate, minCoins, maxCoins, feeTon, currentBalance }

  POST  /api/withdrawals
    Auth: JWT
    Body: { targetAddress: string, amountCoins: number }
    Response: { withdrawal: { id, amountCoins, amountTon, status } }

  GET   /api/withdrawals/history

SECTION 7: LEVEL SYSTEM
================================================================================

7.1  LEVELS

  The game has 13 levels.
  A player's level is determined by the MINIMUM of their two card upgrade
  levels:

    level = min(coinCapacityLevel, miningRateLevel)

  This means a player must upgrade BOTH cards to advance their character level.

7.2  LEVEL EFFECTS

  Level determines:
    - Visual character progression (frontend concern)
    - Eligibility for card upgrades (some upgrades require specific levels)
    - Referral rewards for the referrer

7.3  LEVEL CALCULATION (Pure Function)

  File: src/lib/levelUtils.ts

  function calculateLevel(coinCapacityLevel: number, miningRateLevel: number): number {
    return Math.min(coinCapacityLevel, miningRateLevel);
  }

  After any card upgrade, recalculate level. If level changed upward, trigger
  referral reward for the referrer (see §11).

SECTION 8: SHOP SECTION (CARDS & BOOSTERS)
================================================================================

8.1  CARDS

  Two upgrade cards:

  1. COIN_CAPACITY — Increases tankCapacity.
     Each level increases how many coins the tank can hold during the 1-hour
     fishing session.

  2. MINING_RATE — Increases miningRate (coins per second).
     Each level increases how fast coins generate while fishing.

  Both cards have 13 levels (matching the 13 game levels).

  Card upgrade VALUES are stored in Settings.cardUpgradeValues as JSON:
  {
    "coinCapacity": [100, 200, 350, 500, 750, 1000, 1400, 1800, 2300, 3000, 3800, 4800, 6000],
    "miningRate": [0.0278, 0.0556, 0.0972, 0.1389, 0.2083, 0.2778, 0.3889, 0.5000, 0.6389, 0.8333, 1.0556, 1.3333, 1.6667]
  }
  Index 0 = level 1 value, index 12 = level 13 value.

  NOTE: miningRate values are coins/second. At level 1: ~100 coins/hour.
  At level 13: ~6000 coins/hour (matching max tank capacity).

8.2  CARD UPGRADE RULES

  - Card upgrades are FREE (no coin cost).
  - BUT each level has REQUIREMENTS that must be met before upgrading.
  - Requirements are stored in the CardUpgradeRequirement table and are
    fully admin-manageable.
  - Example requirement types:
    - "level": Player must be at least level N
    - "referrals": Player must have invited N friends
    - "coins_spent": Player must have spent N total coins
    - "task_count": Player must have completed N tasks
    - "login_streak": Player must have a streak of N days

  When upgrading:
    1. Look up requirements for (cardName, nextLevel).
    2. Validate all requirements are met.
    3. Update the card level field.
    4. Update tankCapacity or miningRate to the new level's value.
    5. Recalculate user level = min(coinCapacityLevel, miningRateLevel).
    6. If level changed, trigger referral reward.
    All in prisma.$transaction().

8.3  BOOSTERS

  Two boosters:

  1. MINING_BOOST_1_5X
     - Multiplies mining rate by 1.5 for a set duration.
     - Duration and price (in Stars) defined in Settings.boosterPrices.
     - Only one mining boost can be active at a time.

  2. AUTO_MINING
     - Automatically restarts fishing when collection happens.
     - Duration and price (in Stars) defined in Settings.boosterPrices.
     - While active, the server auto-starts fishing on collect.

  Booster purchase flow:
    1. Validate user has enough starsBalance.
    2. Deduct stars from starsBalance.
    3. Create StarsTransaction (negative amount, source="booster_purchase").
    4. Create ActiveBooster with expiresAt = now + duration.
    5. If MINING_BOOST_1_5X, do NOT restart existing fishing — the boost
       applies on next sync/collect calculation.
    All in prisma.$transaction().

  Booster expiry:
    - On every fishing calculation, check if booster has expired.
    - Expired boosters return multiplier to 1.0.
    - Session cleanup job removes expired booster records.

  File: src/modules/daily-combo/daily-combo.engine.ts

  function evaluateComboAnswer(
    playerItems: number[],    // Array of 4 item IDs in player's order
    correctItems: number[]    // Array of 4 item IDs in correct order
  ): { correctCount: number; reward: number }

  Scoring rules:
    - Compare position by position.
    - Count how many items match in the EXACT same position.
    - 4 correct → 100 coins
    - 3 correct → 75 coins
    - 2 correct → 50 coins
    - 1 correct → 25 coins
    - 0 correct → 0 coins

================================================================================

13.1  MECHANICS

  Players earn coins by logging in daily. The streak resets if they miss a day.

  Streak Rewards (stored in Settings.loginStreakRewards):
    Day 1  → 5 coins
    Day 2  → 10 coins
    Day 3  → 15 coins
    Day 4  → 25 coins
    Day 5  → 35 coins
    Day 6  → 50 coins
    Day 7  → 65 coins
    Day 8  → 80 coins
    Day 9  → 100 coins
    Day 10 → 150 coins

  After Day 10, the cycle repeats from Day 1.

13.2  STREAK LOGIC

  On claim:
    1. Get today's date (UTC, date only, no time).
    2. Get user.lastLoginDate (date only).
    3. If lastLoginDate == today → already claimed, reject.
    4. If lastLoginDate == yesterday → streak continues: loginStreak += 1.
    5. If lastLoginDate is older than yesterday OR null → streak resets: loginStreak = 1.
    6. Determine reward day = ((loginStreak - 1) % 10) + 1.
    7. Look up reward from Settings.loginStreakRewards[rewardDay].
    8. Add reward to user.coins.
    9. Update user.lastLoginDate = today.
    10. Create Action log.

13.3  API ENDPOINTS

  GET   /api/daily-activity/status
    Auth: JWT
    Response: {
      currentStreak: number,
      canClaim: boolean,
      todayReward: number,           // What they'll get if they claim
      streakRewards: [{ day, coins, claimed }],  // Full 10-day display
      resetsAt: string               // Next 00:00 UTC
    }

  POST  /api/daily-activity/claim
    Auth: JWT
    Response: { reward: number, newStreak: number, newBalance: number }

17.1  MECHANICS

    - Top players receive rewards (defined in tournament.rewards JSON).
    - Tournament.isActive set to false.

  1. Stop accepting new HTTP requests.
  2. Stop Telegram bot.
  3. Stop cron jobs.
  4. Wait for in-flight requests (timeout: 10s).
  5. Disconnect Prisma client.
  6. Exit process.
